在这份个人网站作品中，我不仅搭建了前端展示层，也从产品角度完整规划了后端 Restful API，重点围绕两个真实业务场景：简历下载与AI 助手问答。

- 一、围绕用户路径拆解接口，而不是从技术出发随便堆 API
  我先从访问路径出发：
  - 用户在 `首页` 理解我是谁 → 跳转到 `联系我`
  - 在 `联系我` 页面完成简历下载和与 AI 助手对话
    围绕这个路径，我拆出了最小必要但清晰边界的接口：
  - `GET /api/resume`：获取简历元信息（语言版本、文件名、最后更新时间等），方便前端渲染“下载按钮文案 + 提示语”；
  - `GET /api/resume/download`：统一的简历下载入口，支持多语言版本；
  - `POST /api/ai/assistant/messages`：AI 助手对话接口，前端只维护一个 `conversationId` 即可完成多轮对话。
    这样设计的好处是：前端调用简单、职责聚焦、扩展空间明确（比如之后要加下载统计、权限控制，都可以在这几个接口上演进）。
- 二、接口命名与结构遵守 Restful 规范，方便协作与演进
  我刻意避免在路径里加入动词（如 `/downloadResume`），而是用资源名 + 动作语义清晰的子路径：
  - `resume` 作为资源，`/resume` 用于获取元数据，`/resume/download` 用于下载；
  - `ai/assistant/messages` 对应“某个助手下的消息集合”，多轮对话都走这个入口。
    同时，JSON 返回结构统一采用：`success + data + error + traceId`，错误码使用枚举（如 `RESOURCE_NOT_FOUND`、`AI_SERVICE_TIMEOUT`），既便于前端统一处理，也方便后续接入观测与告警体系。
- 三、AI 助手接口体现了我对“产品内 AI 能力落地”的理解
  AI 助手并不是一个单纯的“ChatGPT 小窗”，而是围绕我的履历和项目库定制的“简历问答机器人”：
  - 接口设计上，通过 `conversationId + message + context` 的形式，将用户问题与站内内容（简历模块、作品集模块）绑定；
  - 服务端可以基于这些结构信息构造 Prompt，将项目经历、核心技能、联系方式等关键信息注入模型，使回答更贴合面试场景；
  - 在文档中，我也预留了 `usage` 字段（token 消耗）以及 `AI_SERVICE_UNAVAILABLE` 等错误码，为之后的成本监控和服务降级预留空间。
    这一部分体现的是：我把 AI 当成“产品能力”，而不是简单的功能点——在接口层就为 Prompt 设计、数据注入、监控留好了挂载点。
- 四、为数据化与运维预留空间，而非“一次性 Demo”
  在简历下载和 AI 对话接口之外，我还设计了后续可扩展的：
  - `/api/resume/stats`：简历下载统计接口（总下载量、按语言维度统计等）；
  - `/api/ai/assistant/conversations/{conversationId}`：会话历史查询接口，方便我之后复盘典型问答、优化 Prompt，也可以在后台页面展示给 HR/面试官。
    这些接口在现在的版本中可以是“隐藏功能”，但在文档层面，体现了我在设计之初就考虑了数据分析、运维与后续产品演进，而不是只做前端可见的一层 Demo。

总结：
这个「个人网站接口设计」并不是为了“假装有后端”，而是我在真实业务思路下，对“下载简历 + AI 助手”两个关键场景做的一次完整 API 规划。它体现了我在以下几个方面的能力：

- 需求拆解与用户路径还原（从页面、按钮和真实行为出发设计接口）；
- Restful 规范与接口可演进性（清晰资源边界、统一返回结构与错误码）；
- AI 能力产品化思维（Prompt 链路、上下文数据注入、监控与降级预留）；
- 对数据与运营的前置考虑（下载统计、会话复盘、可视化 Dashboard 的接口基础）。
